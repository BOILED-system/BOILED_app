# 認証機能 設計計画書

## 1. 概要
本機能は、ダンスサークル運営アプリにおけるユーザー認証とセッション管理、およびデータアクセス制御（認可）を提供する機能である。厳密ななりすまし対策よりも、400人規模が迷わずスムーズに利用できる「使い勝手」を最優先とし、**「会員番号」のみを用いた簡易ログイン方式**を採用する。

## 2. ゴール
*   一般メンバーがパスワード管理なしに、自身の会員番号を入力するだけでログインできること。
*   ログイン状態が維持され、再アクセスのたびに番号を入力する手間が省けること。
*   幹部（Admin）や各メンバー（Member）のロールに応じたAPIアクセス制限が行われること。

## 3. 仕様の前提（要求事項から）
*   **認証方式**: ユーザー固有の「会員番号（例: `A001`, `12345` 等）」による単体認証。パスワードや外部IDプロバイダ（Google, LINE）は用いない。
*   **制約事項**: 
    *   本アプリ上でお金（決済）の操作自体は行わず、最終的な消し込み確認はPayPayや銀行口座名義にて行うため、重大ななりすましリスクは許容できる。
    *   他サークルのデータは参照できないよう、Firebaseのセキュリティルール等で保護する。

## 4. 機能の詳細設計（フロントエンド）

### 4.1. ログイン画面 (`/login`)
*   **UI構成**: 
    *   入力フィールド: `会員番号` (テキスト入力または数値入力)
    *   ボタン: `ログイン`
*   **バリデーション**:
    *   未入力チェック
    *   会員番号のフォーマットチェック（サークルのルールに基づく）
*   **動作フロー**:
    1. ログインボタン押下時、入力された会員番号をバックエンドAPI (`POST /api/auth/login`) へ送信。
    2. APIから成功レスポンス（セッショントークン等）を受け取った場合、ブラウザのCookieまたはlocalStorageに保存し、トップページ (`/`) へリダイレクトする。
    3. エラー時、画面上に「会員番号が見つかりません」等のメッセージを表示する。

### 4.2. フロントエンドのセッション管理
*   **状態保持**: React Context または Zustand 等を利用し、グローバルに `currentUser` 状態を保持する。
*   **初期化処理**: アプリのルート（`src/app/layout.tsx` 等）でマウント時にトークンの有無を確認し、あれば自身のアカウント情報を取得 (`GET /api/users/me`) して状態を復元する。
*   **ルーティング制御**: 
    *   未認証ユーザーが保護されたページにアクセスした場合、自動で `/login` へリダイレクトする仕組み（Next.js の Middleware または HOC）を設ける。

## 5. 機能の詳細設計（バックエンド / API）

### 5.1. 認証API群
*   **`POST /api/auth/login`**:
    *   **Request**: `{ "memberId": "A001" }`
    *   **処理**: 
        1. Firestoreの `users` コレクションから、指定された会員番号を持つドキュメントを検索。
        2. 存在する場合、そのユーザーのID（UUID等）をペイロードに含めたJWT（JSON Web Token）を発行する。
        3. JWTを `Set-Cookie` ヘッダ（HttpOnly, Secure）を通じてクライアントに返却するか、レスポンスボディに含めて返す。
    *   **Response**: `200 OK` 
*   **`POST /api/auth/logout`**:
    *   **処理**: クライアント側のCookieを無効化（削除）するレスポンスを返す。
*   **`GET /api/users/me`**:
    *   **処理**: リクエストヘッダ（またはCookie）に含まれるJWTを検証し、該当するユーザーのプロフィール情報とロール（Role）を返却する。

### 5.2. バックエンドの認可（アクセス制御）モデル
*   **カスタムミドルウェア**:
    *   すべての保護されたAPIエンドポイントの前に、JWTを検証してリクエストのコンテキストにユーザー情報をセットするミドルウェア（GoにおけるHTTPインターセプタ）を実装する。
    *   Admin専用API（イベントの作成、集金の作成など）については、「コンテキストのユーザーが `RoleAdmin` であること」をチェックする専用のミドルウェアを用意する。

## 6. データモデル・インフラ設計

### 6.1. Firestore コレクション定義
*   **`users` コレクション**:
    *   ドキュメントID: 自動生成ID または 会員番号を直接用いる
    *   フィールド一覧:
        *   `id`: string (識別子)
        *   `memberId`: string (**NEW** - 会員番号、ログインに使用)
        *   `name`: string (氏名・ニックネーム)
        *   `avatarUrl`: string
        *   `createdAt`: timestamp
        *   `updatedAt`: timestamp
*   **`memberships` コレクション**:
    *   ユーザーとサークル（Circle）の中間テーブル。
    *   フィールド: `userId`, `circleId`, `role` (`ADMIN` or `MEMBER`)

### 6.2. セキュリティルール・セキュリティ考慮事項
*   **JWTの署名**: バックエンド側（Go API）でJWTを発行・検証するための秘密鍵（Secret）は、Google Cloud Secret Manager等で環境変数として安全に管理する。
*   **Firestoreルール**:
    *   現状のアーキテクチャでは、クライアント（フロントエンド）は直接Firestoreへアクセスせず、Go APIを経由する設計となっている。
    *   したがって、Firebase側のSecurity Rulesは「すべて拒否（外部からの直接アクセス禁止）」または「APIサーバー（サービスアカウント）のみ許可」とする。データの保護はGoAPI内の認可ロジックにて担保する。
