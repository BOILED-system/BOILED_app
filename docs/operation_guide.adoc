= サークル運営向け導入・運用ガイド
:toc: left
:toclevels: 2
:sectnums:
:icons: font

このドキュメントでは、本アプリケーション（MVP版）を実際のサークル運営に導入・運用するために必要な作業や確認事項をまとめています。

== 1. 実運用に向けたシステム開発・改修課題

現在のMVPはハッカソン用のダミー実装や未対応機能が含まれているため、実運用前に以下の開発・改修が必要です。

=== 1.1. 認証機能の実装（必須）
現在のシステムは、HTTPヘッダーの `X-User-Id` を利用したダミー認証で動いています。
実際の運用では、なりすましや不正アクセスを防ぐために適切な認証基盤の導入が不可欠です。

* **対応案:** Firebase Authentication や NextAuth.js などの導入
* **作業内容:**
** GoogleログインやLINEログイン、あるいはメールアドレス/パスワード認証の実装
** フロントエンド（fe）でのログイン画面とセッショントークン管理
** バックエンド（be）の各APIエンドポイントでのトークン検証処理の実装

=== 1.2. Firestoreのセキュリティルール設定（必須）
現在は開発のため、誰でも読み書きできる状態（`allow read, write: if true;`）になっている可能性があります。

* **対応案:** 本番用の Firestore Security Rules の作成
* **作業内容:**
** 認証済みユーザーのみがアクセスできるように制限
** ユーザー自身のデータや、所属するサークルのデータのみアクセス・更新できるように権限を制御

=== 1.3. 管理画面・初期設定UIの実装
現在、サークルの作成やメンバーの初期追加はAPI（`curl` コマンド等）経由でのみ可能となっています。

* **作業内容:**
** サークル作成、メンバー招待（URL共有やID追加など）UIの実装
** サークル管理者（admin）向けの権限管理UIの実装

=== 1.4. 未接続機能の結合
* **作業内容:**
** 「お知らせ単位の出欠登録（出席機能）」のAPIエンドポイントが未接続となっているため、バックエンドのルーターへの紐付けとフロントエンドからの連携を実装

=== 1.5. ロール（権限）の分離とUIの出し分け（必須）
現在、誰でもイベントや清算を作成できる状態（フロントエンドで制限がない状態）になっています。運営を安全に行うため、管理者（幹部）と一般メンバーの役割分担が必要です。

* **対応案:** Role-Based Access Control (RBAC) の導入
* **作業内容:**
** DB（Firestoreの `memberships` コレクションなど）で、各ユーザーに `admin`（管理者）または `member`（一般）のロールを付与
** フロントエンド（fe）にて、ログイン中のユーザーが `admin` の場合のみ「イベント作成」「お知らせ作成」「清算作成」ボタンや編集画面を表示
** バックエンド（be）にて、作成・編集APIリクエスト時に該当サークルの `admin` 権限を持っているかチェックするミドルウェアの追加

---

== 2. 本番環境の構築とデプロイ設定

アプリを一般公開してメンバーに使ってもらうためのインフラ構築作業です。

=== 2.1. インフラのプロビジョニング（GCP & Vercel）
`infra/terraform` に用意されているコードを活用し、各種リソースを作成します。

* **対応案:**
** **GCPプロジェクトの作成:** 本番用の独立したプロジェクトを用意
** **Firestoreの有効化:** 本番用のデータベース（Nativeモード）を設定
** **Gemini API Keyの取得と設定:** `Secret Manager` に安全に登録
** **Cloud Run デプロイ:** バックエンド API（be）コンテナのビルドとデプロイ
** **Vercel デプロイ:** フロントエンド（fe）の接続先API URL（`NEXT_PUBLIC_API_BASE_URL`）をCloud RunのURLに向けてデプロイ

=== 2.2. ドメインとSSL通信の設定（推奨）
* **作業内容:**
** Vercel側でカスタムドメイン（例: `app.your-circle.com`）の設定
** 必要に応じて API（Cloud Run）側のカスタムドメイン設定

---

== 3. サークル運営としての運用準備

システムが稼働した後に、幹事やメンバーが行う運用の取り決めです。

=== 3.1. マスターデータの整備
* **対応案:**
** サークル情報（名前、ロゴ、概要）の登録
** メンバーアカウントの作成・招待と、初期情報の登録
** 直近のイベント（例: 新歓公演、深夜練、合宿など）や過去の重要なお知らせ（例: スタジオ利用のルール等）の入力。これらを事前に入力しておくことで、AIチャットが「深夜練のスタジオはどこ？」といった質問に的確に回答できるようになります。

=== 3.2. 清算（割り勘）ルールの明確化と周知
本アプリには支払い・清算機能が含まれていますが、実際の送金はPayPayや銀行振込などの外部サービスで行われます。

* **確認事項:**
** 送金先ID（PayPay IDや銀行口座情報）の正確な登録
** 「支払い報告」ボタンを押すタイミングのルール化（送金完了後すぐ、など）
** 未払い者へのリマインド運用ルールの決定

=== 3.3. バックアップとデータ管理
* **作業内容:**
** Firestoreデータの定期バックアップ設定（GCPのマネージドエクスポート機能など）
** 間違って削除してしまったイベントや支払い情報の復旧フローの確認

---

== 4. まとめ：最短で稼働させるためのステップ

1. **認証の組み込み（Firebase Auth等）** と **Firestoreセキュリティルールの設定** を行う（※最優先・セキュリティ直結）。
2. `infra/terraform` を用いて **GCP環境** を構築し、バックエンドを Cloud Run にデプロイ。
3. Vercelなどへフロントエンドをデプロイし、動作確認。
4. サークルの実データ（初期メンバー、直近のイベント等）を投入して運用開始。
