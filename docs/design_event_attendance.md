# イベント・出欠機能 設計計画書

## 1. 概要
本機能は、ダンスサークルにおける各種イベント（新歓公演などの全体イベントから、深夜練などの特定ジャンル・ナンバー限定の練習まで）の作成・表示と、メンバーの出欠（RSVP）登録を管理するコア機能である。約400名規模の運用に耐えうるよう、対象者を「ジャンル（ナンバー）」単位で絞り込む機能と、柔軟な出欠ステータス管理を提供する。

## 2. ゴール
*   **全体/単発イベント**: 定期イベント同様に、特定の「ジャンル（対象者）」に絞ったイベント作成・表示制御が行えること。
*   **定期・練習イベント**: 現状対象者を絞れない「定期・練習」タブにおいても、特定のジャンルやナンバーに紐付けた作成・表示ができること。
*   **出欠登録**: GO(参加), NO(不参加), LATE(遅刻), EARLY(早退)の4ステータスと自由記述メモを登録・変更できること。
*   将来的な拡張（香盤表との連携、ナンバーごとの出欠集計）を見据えたデータ構造を採用すること。

## 3. 機能の詳細設計（フロントエンド）

### 3.1. イベント作成・編集UI (`/events/new`, `/events/[id]/edit`)
*   **対象者指定UIの刷新**:
    *   現状の「ユーザーを1人ずつ選択するUI」を廃止。
    *   代わりに「対象ジャンル（ナンバー）」をプルダウンやマルチセレクトで選択するUIへ変更。
    *   「全体（All）」を選択した場合はサークル全員が対象となる。

### 3.2. カレンダー・一覧表示 (`/events`, `/calendar`)
*   **表示制御**:
    *   ログインユーザー（`currentUser`）が所属しているジャンル（ナンバー）に紐づくイベント、または「全体」イベントのみをリストおよびカレンダー上に表示する。
    *   関係のない他ジャンルの深夜練等はカレンダーを圧迫するため意図的に非表示（または薄く表示するトグル）とする。

### 3.3. イベント詳細・出欠パネル (`/events/[id]`)
*   **RSVPパネル**:
    *   対象ユーザーのみに表示される。
    *   [出席] [遅刻] [早退] [欠席] の4ボタンと、メモ（テキストフィールド）を入力して送信。
    *   遅刻・早退を選択したメンバー向けには、メモ欄に「何時から参加か／何時に抜けるか」の入力を促すプレースホルダを表示する。
*   **管理者・ジャンルリーダー（Admin）向けの表示**:
    *   出欠状況（出席〇人、欠席〇人）の集計と、メンバーそれぞれの出欠ステータス一覧を表示。

## 4. 機能の詳細設計（バックエンド / API）

### 4.1. データモデルの拡張（ジャンル/ナンバー）
約400名規模の管理を実現するため、ユーザーを束ねる「グループ（枠組み）」の概念をシステムに導入する。

*   **`genres` または `numbers` コレクションの追加** (例: `genres`):
    *   `id`: string
    *   `circleId`: string
    *   `name`: string (例: "HipHop", "Jazz", "M1_オープニング")
*   **`user_genres` (中間テーブル) の追加**:
    *   どのユーザーがどのジャンルに所属しているかを管理。
    *   `userId`: string
    *   `genreId`: string

### 4.2. イベントデータモデル (`events`, `practice_series` 等) の改修
*   **`targetGenreIds` の追加**:
    *   既存の `rsvpTargetUserIds: string[]`（個別ユーザーID指定）から、`targetGenreIds: string[]`（対象のジャンルID指定）へデータ構造を移行。
    *   空配列の場合は「全体対象（サークル全員）」と扱うルールとする。

### 4.3. バックエンドAPIの主な改修ポイント
*   **`GET /api/events` / `GET /api/practices`**:
    *   **フィルタリングロジック**:
        1.  リクエスト元のユーザーID (`currentUser.id`) を取得。
        2.  そのユーザーが所属している `genreId` の一覧をDBから取得。
        3.  イベントの `targetGenreIds` が空（＝全体）、またはユーザーの所属する `genreId` が含まれているイベントのみを抽出してクライアントへ返す。
*   **`POST /api/events`**:
    *   ペイロードに `targetGenreIds` を受け取り、Firestoreに保存する処理を追加。
*   **`POST /api/events/:eventId/rsvp`**:
    *   （既存のままでほぼ流用可能）`status` (GO, NO, LATE, EARLY) と `note` を受け取り保存。
    *   ユーザーが本当にそのイベントの対象者であるか（所属ジャンルのチェック）を入れるとより堅牢。

## 5. 将来の拡張に向けた検討

*   **香盤表（出番表）との連携**:
    イベント自体は1つ（例「新歓公演リハーサル」）でも、その中で「M1（1曲目）」の練習時間、「M2（2曲目）」の練習時間が分かれている場合がある。
    現状はイベントを分ける運用でカバーするが、将来的には「1つのイベント内に複数のセッション（ナンバーごとの出番）」を持ち、セッションごとに出欠（遅刻・早退）を紐付けるような複雑なデータ構造への拡張余地を残しておく。
